Code Reviews

In the "real world", most code reviews are done by the other members of your development team in response to a pull request from your working branch in the team Git repository. You will follow a similar workflow for your lab assignments.

Create a repository on GitHub

  • Create one private repository for all your lab assignments in this class.
    GitHub now offers free private repositories with up to three collaborators.
  •  Add your instructor and your lab partner as collaborators.
    You can restrict them to read-only priveledges if you wish.
  • You will create two branches for each lab assignment, a beta branch and a production branch.
    The weekly lab instructions will provide guidance for naming the branches.

Lab assignment work-flow

Working with Git and sending a pull request

  1. To facilitate grading, create two branches for each lab assignment.
    For example, branches for lab 1 might be:
    • Lab1-Master
    • Lab1-Beta
  2. Put any starting files in Lab1-Master.
    Starting files might be:
    • Files provided by your instructor
    • The finished version of your previous lab assignment
    • An empty project with default files created by the IDE
  3.  Do your lab work, making frequent commits to the Lab1-Beta branch.
  4. When your beta version is done, send a PR with Lab1-Master as the base being compared to Lab1-Beta.
    • Reminder: you can only send a PR if the code on the two branches is different.
  5. Make revisions to your code on Lab1-Beta after getting a review from your lab partner.
  6. Merge Lab1-Beta into Lab1-Master.
    Lab1-Master is now the production version for this lab assignment and this is what you will submit to your instructor.

Doing a code review

When you get a PR from your lab partner, you will follow these steps to revie their code:
  1. Clone your lab partner's repository to your development machine, if you haven't already done so.
  2. Pull the branch specified in the PR
  3. Run the code to test it. Check to see if it meets all the requirements for this lab assignment.
  4. Inspect the code to check for good coding practices
  5. Provide the review in the PR comments section on GitHub

For additional guidance, read my step-by-step instrucitons with screen-shots.

Deadlines

  • PRs need to be sent by the due date shown on Moodle or in the syllabus.
    • Your lab partner is not obligated to give you a review if your PR is sent late.
  • Code reviews need to be given by the date shown on Moodle or in the syllabus.
    • Code reviews given after the due date for the lab assignment won't be accepted for grading.

FAQ

  • What if my lab partner doesn't send me a PR?
    1. Attempt to contact your lab partner and ask them to send a PR.
    2. If your lab partner has still not sent a PR, contact your instructor.
      • Your instructor may be able to send you a PR for an alternate project to review.
  • What if my lab partner doesn't review my code?
    • If you feel confident about your beta version, you can just go ahead and merge your code and submit it.
    • If you still want a review, you can contact your instructor to either give you a review or arrange for someone else to give you a review.
